手工测试如何过渡到自动化测试

发表于 2022-02-06 分类于 测试

手工测试与自动化测试的区别

手工测试：顾名思义就是测试人员自己手动执行所有测试步骤，以便在应用程序发布之前查找错误。

自动化测试：测试人员编写自动化脚本以自动化工具自动执行测试步骤。测试人员将测试脚本存储起来作为资产来进行重用，从而为重复执行节省大量时间和精力。

为什么要从手工测试转到自动化测试

自动化测试可以克服许多手工测试中遇到的挑战，在敏捷项目中尤为重要。

1、测试可重用性

自动化测试用例和测试套件可以在不同的测试周期和测试环境中多次重用。因此，每次应用程序更改时，您都可以运行自动回归测试套件来检查回归错误，从而避免重复手动执行回归测试。这是自动化降低操作故障风险的最重要优势之一。

2、更高的测试覆盖率

由于测试是自动执行的，因此有更多时间专注于新的测试方案，并编写更自动化的测试用例来验证和确认受测应用程序 （AUT）。可以自由地进行更多的探索性测试，以确保产品质量。

自动测试也可以在不同的平台和设备上同时或并行执行。执行的测试越多，意味着可能会发现更多的回归错误，并确认更多的业务需求。

3、更早的错误检测和反馈

在生产环境中发现的 Bug 在时间、金钱和精力等方面修复成本更高。这就是为什么团队正在向左移动，更早地发现错误以最大限度地降低成本。

如果自动化测试做得好，可以在开发阶段开始后立即开始。开发人员可以在将代码提交到版本控制系统之前在其本地计算机上执行单元测试。

单元测试也可以集成到 CI 管道中。提交代码后，将触发管道以运行自动化单元测试，然后运行集成或系统测试。通过这种方式，开发人员可以更快地接收反馈，检测和修复错误，防止bug潜入后期阶段。

4、超越手工测试

测试自动化可以完成手动测试可能难以实现的工作，特别是对于需要大量精力和资源（如性能测试）的特定测试。它可以模拟来自不同平台和设备对AUT服务器端的数千个同时请求，这几乎不可能手动完成。

在并行执行方面，自动化工具的工作速度比人类快，精度更高。机器不会像人类那样感到无聊或疲惫，可以避免类似人类的错误。因此，测试结果是可靠且快速接收的。（PS：要记住的一件事是，长远来看，自动化测试是有效的，但是自动化测试初始成本较高，需要时间和设备将所有内容落实到位，才会带来可观的效果并降低成本效益比）。

什么情况不值得开展自动化测试？

以下四种情况，要开展自动化测试前请慎重、慎重、慎重考虑

  • 短期的项目
  • 开发在4-6月内完成的简单应用程序
  • 不经常运行测试
  • 测试许多手工执行起来更容易的复杂方案

从手工测试过渡到自动化测试

Step1、设计自动化测试策略

许多团队通常都很挣扎，因为他们没有面向目标的测试自动化策略。它使所有团队成员都在同一页面上。它告知并传达预先定义的目标和计划，每个人都可以从中了解他们所做的事情是对还是错，并调整下一步行动。

如果没有清晰的愿景战略，团队很容易改变方向，选择错误的解决方案，并且无法以有助于实现业务价值的方式使用自动化测试。

自动化测试就像构建并拥有自己的软件来测试其他应用程序一样。它应该以与软件开发相同的方式以清晰的愿景和计划进行管理。

Setp2、丰富自动化测试基础

了解测试自动化的所有基础知识至关重要，从敏捷和 DevOps 方法、框架构建、自动化金字塔、架构、自动化方法到报告和指标。没有良好的基础，团队几乎无法执行其战略。

此外，还应该检查特定的自动化解决方案或测试工具，它们需要哪些技能，它们解决了什么问题，以及它们在实践中是如何做到的。并保持学习过程不间断。

许多团队只是在一段时间后停止更新其知识库。随着新技术、新趋势和新解决方案的出现，行业发展迅速，停止学习和培训只能意味着落后于竞争对手。

Step3、 编程技能

许多测试自动化解决方案需要编程技能来构建框架和编写自动化测试脚本。为了克服这些障碍，许多团队的目标是使用配备低代码功能的现成测试工具。

但是，没有编程技能，测试人员只能使用内置的关键字、录制和播放功能，或者具有直观 UI 的手动模式来创建自动测试。事情会永远这么简单吗？

从长远来看，要编写更复杂的测试用例、修改脚本、构建自定义关键字、维护测试或调试失败的测试，团队中至少应该有一个熟悉编码和编程的自动化测试人员或一个开发人员来支持。

因此，如果团队想要从手动测试切换到自动测试，请考虑雇用具有编程背景的测试人员，或者让团队成员学习和提高他们的编码技能。

Step4、 选择适合自己的测试工具

工具选择可能非常棘手，因为市场上有数千种可用选项。仅根据其他人的成功来选择他们的工具。但是，一个工具对其他工具有效并不一定意味着它对你有同样的作用。

考虑团队的特定需求和资源。也就是说，这里有一些问题需要评估和选择最合适的解决方案：

  • 想解决什么问题？确定要应用自动测试的 AUT（Web、桌面或移动应用程序）以及所需的功能（用于测试创建、执行、报告等）。
  • 谁使用该工具？ 如果他们是手动测试人员，那么低代码解决方案将更合适。
  • 能否融入团队的现有流程和工具链？寻找具有本机集成的自动化工具，以减少变通方法的时间。
  • 是否面向未来？为避免从一个工具切换到另一个工具，请考虑团队的可伸缩性，并选择可以满足未来需求的工具。
  • 预算是多少？ 开源是免费的，但需要时间来设置和开发测试自动化工具，而商业工具可以快速启动且易于使用。
  • 从哪里可以获得支持？良好的客户支持或活跃的社区可以在处理技术问题时真正帮助您并节省时间。

提示：从手动起点开始，最好选择一个现成的工具，它具有低代码和脚本选项，如Katalon Studio。它将为您节省大量时间来构建框架和功能，这是启动测试自动化时压力最大的部分。

借助”录制”和”回放”等低代码功能，手动测试人员可以快速开始自动执行测试用例并捕获测试对象。之后，当拥有更多的编码经验时，他们可以继续编写测试脚本，构建更多的自定义关键字，并利用更高级的功能。

选择工具后，请记住进行概念验证。在工具上运行快速冲刺，让所有团队成员参与进来。然后，在冲刺 （sprint） 结束时，收集所有反馈并评估该工具是否能够满足团队的所有标准要求。

Setp5、 小demo运行，自信扩展

从小项目的特定小目标开始，例如在一段时间内达到某个自动化测试用例。保持错误小，从中吸取教训，并逐步扩大团队测试自动化的规模。不要因为过于雄心勃勃致使事情复杂化而延迟过渡。

The End

从手动自动化转向测试自动化可能是一个困难的长期过程。没有仔细的规划，缺乏所需的技能组合以及不合适的解决方案是导致团队挣扎的一些最常见的根本原因。欢迎在下边的讨论区我们一起交流。

2022年软件测试趋势

发表于 2022-02-06 分类于 测试

技术呈指数级和前所未有的变化影响了组织开发、验证、交付和操作软件的方式。

因此，这些组织必须通过寻找解决方案来不断创新和改造自己，以优化实践和工具，以快速开发和交付高质量的软件。

软件测试约占项目总工作量的30%，是更改和改进的重要焦点。测试实践和工具需要不断发展，以应对在系统、环境和数据日益复杂的情况下实现**”快速测试”**的挑战。

在下面介绍了软件测试的主要趋势，其中许多趋势在过去几年中已经出现。我们观察到，敏捷和DevOps、测试自动化、用于测试的人工智能和API测试自动化是2022年和未来几年最引人注目的趋势。

1、敏捷和Devops

敏捷已经视为对快速变化的需求的回应，并将DevOps视为对速度需求的响应。

DevOps 涉及实践、规则、流程和工具，有助于集成开发和运营活动，从而缩短从开发到运营的时间。DevOps已成为组织广泛接受的解决方案，这些组织正在寻找缩短从开发到交付和运营的软件生命周期的方法。

敏捷和DevOps的采用有助于团队更快地开发和交付高质量的软件，这反过来又被称为”速度的测试”。这种采用在过去五年中引起了很大的兴趣，并且在未来几年也将继续加强。

2、测试自动化

为了有效地实施 DevOps 实践，软件团队不能忽视测试自动化，因为它是 DevOps 流程的基本要素。

需要找到机会，用自动化测试取代手动测试。由于测试自动化被认为是DevOps的重要瓶颈，因此至少大多数回归测试都应该是自动化的。

鉴于DevOps的普及以及测试自动化未得到充分利用的事实，只有不到20%的测试是自动化的，因此在组织中增加测试自动化的采用空间很大。应该出现更先进的方法和工具，以便在项目中更好地利用测试自动化。

3、 API和服务自动化测试

将客户端和服务器分离是设计 Web 和移动应用程序的当前趋势。

API 和服务在多个应用程序或组件中重用。反过来，这些更改要求团队独立于使用它们的应用程序测试 API 和服务。

当跨客户端应用程序和组件使用 API 和服务时，测试它们比测试客户端更有效和高效。趋势是对API和服务测试自动化的需求继续增加，可能超过最终用户在用户界面上使用的功能。

为 API 自动化测试提供正确的流程、工具和解决方案比以往任何时候都更加重要。

4、用于测试AI和ML

尽管应用人工智能和机器学习（AI / ML）方法来解决软件测试中的挑战在软件研究社区中并不新鲜，但AI / ML的最新进展以及大量可用的数据为在测试中应用AI / ML提供了新的机会。

然而，AI/ML在测试中的应用仍处于早期阶段。扔在寻找在AI / ML中优化其测试实践的方法。

AI/ML 算法旨在生成更好的测试用例、测试脚本、测试数据和报告。预测模型将有助于决定在何处、做什么以及何时进行测试。智能分析和可视化支持团队检测故障，了解测试覆盖范围，高风险区域等。

希望在未来几年看到AI / ML在解决质量预测，测试用例优先级，故障分类和分配等问题中的更多应用。

5、移动测试自动化

随着移动设备的功能越来越强大，移动应用程序开发的趋势继续增长。

要完全支持 DevOps，移动测试自动化必须是 DevOps 工具链的一部分。然而，目前移动测试自动化的利用率非常低，部分原因是缺乏方法和工具。

移动应用的自动化测试趋势持续增加。这种趋势是由缩短上市时间的需求以及用于移动测试自动化的更高级方法和工具所驱动的。

6、测试环境和测试数据

物联网（IoT）的快速增长）意味着更多的软件系统在众多不同的环境中运行。这给测试团队带来了确保正确级别的测试覆盖率的挑战。事实上，在敏捷项目中申请测试时，缺乏测试环境和数据是最大的挑战。

将看到在提供和使用基于云的容器化测试环境方面的增长。AI/ML在生成测试数据方面的应用和数据项目的增长，是解决测试数据不足的一些解决方案。

什么是Cookie？

Cookie 是由网络服务器存储在用户硬盘上的文本文件中的一小段信息。Web 浏览器稍后将使用此信息从该计算机检索信息。

通常，Cookie包含用于在不同网页之间进行通信的个性化用户数据或信息。

为什么使用cookie？

Cookies只不过是记录用户的身份，用于跟踪用户在整个网站页面中的导航位置。但是Web 浏览器和 Web 服务器之间的通信是无状态的。

例：

如果正在访问域”http://www.example.com/1.html"，则Web浏览器将仅向 example.com Web服务器查询第1.html页。

下次，如果将页面键入为”http://www.example.com/2.html"，则新请求将发送到 example.com Web服务器以发送2.html页面，并且Web服务器不知道有关上一页1.html的任何信息。

如果想要此用户与 Web 服务器通信的先前历史记录，该怎么办？需要在某个位置维护 Web 浏览器和 Web 服务器之间的用户状态和交互。这就是饼干进入画面的地方。Cookie 用于维护用户与网络服务器的交互。

Cookie是如何工作的？

Cookie用于维护在HYYP协议上 Web 上交换信息的文件。

有两种类型的 HTTP 协议。无状态 HTTP 和有状态 HTTP 协议。无状态 HTTP 协议不保留以前访问过的网页历史记录的任何记录。虽然有状态 HTTP 协议确实保留了以前的 Web 浏览器和 Web 服务器交互的一些历史记录，但 Cookie 使用此协议来维护用户交互。

每当用户访问使用cookie的网站或页面时，该HTML页面中的小代码（通常，调用某些语言脚本来编写cookie，如JAVAScript，PHP，Perl中的cookie）会在用户的计算机上写入一个文本文件，称为cookie。

下面一段示例代码如何设置写一个cookie，可以放在任何HTML页面上：

Set-Cookie: NAME=VALUE; expires=DATE; path=PATH; domain=DOMAIN_NAME;

当用户稍后访问同一页面或域时，将从磁盘读取此 Cookie，并用于标识该域上同一用户的第二次访问。过期时间是在写入 Cookie 时设置的。此时间由要使用 Cookie 的应用程序决定。

通常，在用户计算机上写入两种类型的Cookie

会话 Cookie：此 Cookie 处于活动状态，直到调用该 Cookie 的浏览器打开。当我们关闭浏览器时，此会话cookie将被删除。有时，可以将 20 分钟的会话设置为使 Cookie 过期。

持久性 Cookie：这些是永久写入用户计算机上并持续数月或数年的 Cookie。

Cookie存储在哪里？

当任何网页应用程序写入 Cookie 时，它会保存在用户硬盘驱动器上的文本文件中。存储 Cookie 的路径取决于浏览器。不同的浏览器将 Cookie 存储在不同的路径中。

Cookie 存储位置示例：

Edge： “C：\Users\username\AppData\Roaming\Microsoft\Windows\Cookies”。

**Windows 7：”**C：\Users\username\AppData\Roaming\Microsoft\Windows\Cookies\Low”。

**Windows 8 和 Windows 10：”**C：\Users\username\AppData\Local\Microsoft\Windows\INetCookies”。

在这里，”默认用户”可以替换为您登录的当前用户，如”管理员”，或用户名，如”admin”等。通过浏览浏览器选项可以轻松找到 Cookie 路径。

在Mozilla Firefox浏览器中，您甚至可以在浏览器选项本身中看到cookie。打开 Mozilla 浏览器，按”打开菜单”按钮？”网络开发人员”？”存储检查器”或按组合”Shift + F9”。

在谷歌浏览器中，您可以通过在地址栏中输入”chrome://settings/content/cookies”来查找cookie。也可以使用浏览器控制台访问 Cookie：按 F12 – > 应用程序 – >存储 – > Cookie。

Cookie是如何存储的？

以 rediff.com 在 Mozilla Firefox 浏览器上编写的 Cookie 为例。在 Mozilla Firefox 浏览器上，当您打开 rediff.com 页面或登录您的 Rediffmail 帐户时，您的硬盘上会写入一个 Cookie。

要查看此 Cookie，只需单击上述路径中提到的”显示 Cookie”按钮即可。单击此 Cookie 列表下的 Rediff.com 站点。

可以看到 Rediff 域中写入的不同 Cookie 具有不同的名称。

Site: Rediff.com Cookie name: RMID
Name: RMID (Name of the cookie)
Content: 1d11c8ec44bf49e0… (Encrypted content)
Domain: .rediff.com
Path: / (Any path after the domain name)
Send For: Any type of connection

Cookie可以使用在哪里？

实现购物车： Cookie 用于维护在线订购系统。如果用户将某些产品添加到购物车中，并且由于某种原因用户这次不想购买这些产品并关闭浏览器窗口，该怎么办？Cookie记住用户想要购买的东西。下次同一用户访问购买页面时，他可以看到他在上次访问期间添加到购物车中的所有产品。

个性化网站： 当用户访问某个页面时，系统会询问他们不想访问或显示哪些页面。用户选项存储在cookie中，直到用户在线，这些页面不会显示给他。

用户跟踪： 跟踪特定时间的在线唯一身份访问者数量。

营销： 一些公司使用Cookie在用户机器上显示广告。Cookie 控制这些广告。何时以及应显示哪个广告？用户的兴趣是什么？他们在网站上搜索哪些关键字？所有这些东西都可以使用cookie进行维护。

用户会话： Cookie 可以使用用户 ID 和密码跟踪用户与特定域的会话。

Cookie的缺点

用户禁用：使用Cookie是维护用户交互的好方法。但是如果用户在编写任何Cookie或完全禁用Cookie之前设置了浏览器选项以警告，则包含Cookie的网站将被完全禁用，并且无法执行任何操作，从而导致网站流量损失。这可以在浏览器设置中禁用/启用。

例如：对于谷歌浏览器，需要转到设置->高级->内容设置->Cookie。在那里，可以应用 Cookie 策略的一般规则，或为各个网站设置规则。

Cookie 过多：如果在每个页面导航上写入了过多的 Cookie，并且用户在编写 Cookie 之前打开了警告选项，这甚至可能会使用户离开您的网站。

安全问题：有时，用户的个人信息存储在Cookie中，如果有人入侵Cookie，那么黑客可以访问您的个人信息。即使是损坏的 Cookie 也可能被不同的域读取，并导致安全问题。

敏感信息：某些网站可能会在Cookie中写入和存储敏感信息，出于隐私考虑，这不应该被允许。

Cookie的测试用例

#1）请确保在您的设计文档中，Cookie 中未存储任何个人或敏感数据。

#2） 如果除了将敏感数据保存在 Cookie 中之外别无选择，请确保存储在 Cookie 中的数据以加密格式存储。

#3）确保在被测的网站上没有过度使用 Cookie。如果浏览器更频繁地提示输入 Cookie，则过度使用 Cookie 会惹恼用户，这可能会导致网站流量损失并最终导致业务损失。

#4）从您的浏览器设置中禁用 Cookie。如果您在网站上使用 Cookie，则禁用 Cookie 将无法使用您网站的主要功能。然后尝试访问受测网站。导航到该网站，查看是否向用户显示适当的消息，例如”为了使本网站顺利运行，请确保在您的浏览器上启用了Cookie”。任何页面都不应因禁用 Cookie 而崩溃。在执行此测试之前，请确保关闭所有浏览器并删除所有以前编写的cookie。

#5）接受/拒绝某些 Cookie：检查网站功能的最佳方法是不接受所有 Cookie。如果您在Web应用程序上编写10个Cookie，则随机接受一些Cookie，例如接受5个Cookie并拒绝5个Cookie。

要执行上述测试用例，您可以将浏览器选项设置为每当在磁盘上写入Cookie时都会提示。在此提示窗口中，您可以接受或拒绝 Cookie。尝试访问网站的主要功能，并查看页面是否崩溃或数据是否损坏。

#6）删除 Cookie：允许网站编写 Cookie，然后关闭所有浏览器并手动删除受测网站的所有 Cookie。访问网页并检查网页的行为。

#7）损坏 Cookie：损坏 Cookie 很容易。可以在记事本中手动编辑 Cookie，并将参数更改为一些模糊的值，例如更改 Cookie 内容、Cookie 名称或 Cookie 的到期日期，并查看网站功能。

在某些情况下，损坏的 Cookie 允许读取任何其他域的数据。对于网站 Cookie，不应发生这种情况。请注意，如果 Cookie 是由一个域编写的，如 rediff.com写的Cookie 另一个yahoo.com域无法访问它，除非 Cookie 已损坏并且有人试图破解 Cookie 数据。

#8 ）检查从您的 Web 应用程序页面中删除 Cookie：有时，由域编写的 cookie 表示 rediff.com 可能由同一域删除，但由该域下的不同页面删除。如果您正在测试某操作跟Web 门户，则这是一般情况。操作跟踪或购买跟踪像素放置在操作网页上，当用户发生任何操作或购买时，写入磁盘上的Cookie将被删除，以避免从同一Cookie记录多个操作。检查到达您的操作或购买页面是否正确删除了Cookie，并且不再记录来自同一用户的无效操作或购买。

#9）在多个浏览器上进行 Cookie 测试：这是检查您的 Web 应用程序页面是否按预期在不同的浏览器上正确编写 Cookie 以及网站是否使用这些 Cookie 正常工作的最重要情况。您可以在最常用的浏览器（如Chrome，Edge，Mozilla Firefox，Netscape，Opera等）上测试您的Web应用程序。

#10）如果使用 Cookie 来维护任何用户的日志记录状态，请使用某些用户名和密码登录到您的 Web 应用程序。

在许多情况下，您可以直接在浏览器地址栏中看到登录的用户 ID 参数。将此参数更改为其他值，例如，如果以前的用户 ID 为 100，则将其设置为 101，然后按 Enter。应向用户显示正确的访问消息，并且用户不应能够看到其他用户的帐户详细信息。

#11）通过在 Cookie 文件或浏览器控制台中检查 Cookie 类型和到期日期来验证 Cookie 是否持久（是否符合要求）

#12）验证是否根据要求相应地设置了到期日期。在某些情况下，通过使用应用程序（例如刷新会话）来检查Cookie到期日期是否更新至关重要。这可以在浏览器控制台或 cookie 文件本身中进行检查。

请注意，手动解码cookie不是最佳方法，依赖浏览器数据更加容易。

#13）如果某些 Cookie 是特定于用户的，请务必确保在其他用户登录应用程序时将其删除或直接忽略，除非另有说明。

#14）针对多环境站点的特定测试：检查是否在所有环境中都可接受相同的 Cookie。问题可能是由 Cookie 路径中使用通配符（所谓的超级 cookie）引起的。如果这是允许它的要求，则由于使用了不同的加密密钥（例如，对于 .Net，除非另有指定，否则它通常是唯一的计算机密钥），因此可能会导致一些访问问题。

什么是安全测试？

安全测试是一个检查机密数据是否保密的过程（即，它不会暴露给不适合它的个人/实体），并且用户只能执行他们有权执行的任务。

例用户不应能够拒绝其他用户使用网站的功能，或者用户不应能够以意外的方式更改Web应用程序的功能等。

安全测试中的术语

“漏洞”

这是 Web 应用程序中的一个弱点。”漏洞”的原因可能是由于应用程序中的错误，注入（SQL /脚本代码）或病毒的存在。

“网址操作”

某些 Web 应用程序在 URL 中的客户端（浏览器）和服务器之间传递其他信息。更改 URL 中的某些信息有时可能会导致服务器出现意外行为，这称为 URL 操作。

“SQL注入”

这是通过 Web 应用程序用户界面将 SQL 语句插入到某个查询中，然后由服务器执行的过程。

“XSS（跨站点脚本）”

当用户在 Web 应用程序的用户界面中插入 HTML/客户端脚本时，此插入对其他用户可见，称为 XSS。

“欺骗”

欺骗是创建类似恶作剧的网站和电子邮件。

web安全测试的方法

密码破解

Web应用程序的安全测试可以通过”密码破解”开始。为了登录到应用程序的私人区域，人们可以猜测用户名/密码或使用一些密码破解工具。常见的用户名和密码列表以及开源密码破解程序一起可用。

如果 Web 应用程序不强制实施复杂密码（例使用字母、数字和特殊字符或至少具有所需数量的字符），则破解用户名和密码可能不会花费很长时间。

如果用户名或密码存储在 Cookie 中而未经过加密，则攻击者可以使用不同的方法来窃取 Cookie 以及存储在 Cookie 中的信息（如用户名和密码）。

测试人员应检查应用程序是否传递查询字符串中的重要信息。当应用程序使用 HTTP GET 方法在客户端和服务器之间传递信息时，会发生这种情况。

信息通过查询字符串中的参数传递。测试人员可以修改查询字符串中的参数值，以检查服务器是否接受它。

通过HTTP GET请求将用户信息传递到服务器进行身份验证或获取数据。攻击者可以操纵从此 GET 请求传递到服务器的每个输入变量，以获取所需信息或损坏数据。在这种情况下，应用程序或 Web 服务器的任何异常行为都是攻击者进入应用程序的入口。

网站测试要考虑的点

发表于 2022-02-05 分类于 测试

网站本质上都是B/S架构的应用程序-具有service和browser。

测试时应考虑 HTML 页、TCP/IP 通信、Internet 连接、防火墙、在网页中运行的应用程序（如小程序、javascript、插件应用程序）和在服务器端运行的应用程序（如 CGI 脚本、数据库接口、日志记录应用程序、动态页面生成器、asp 等）之间的交互。

此外，还有各种各样的服务器和浏览器，每种服务器和浏览器都有不同的版本。它们包括它们之间在连接速度变化，快速变化的技术以及多个标准和协议方面的小但有时显着的差异。对网站的测试的最终结果可以成为一项重大的持续工作。

测试网站需考虑的因素

  • 服务器上的预期负载是多少（例如，单位时间内的命中数）？
  • 在每种负载条件下（如 Web 服务器响应时间、数据库查询响应时间）下需要什么样的性能？
  • 性能测试需要什么样的工具（例如Web负载测试工具，其他可以适应的内部工具，Web机器人下载工具等）？
  • 谁是目标受众？他们将使用哪种浏览器？他们将使用哪种连接速度？它们是组织内部的（因此可能具有高连接速度和类似的浏览器）还是Internet范围的（因此具有各种连接速度和浏览器类型）？
  • 客户端期望获得什么样的性能（例如，页面应该以多快的速度显示，动画、小程序等的加载和运行速度应该有多快）？
  • 是否允许服务器和内容维护/升级的停机时间？如果是这样，那么多少钱？
  • 需要什么样的安全性（防火墙，加密，密码等），以及它应该做什么？如何对其进行测试？
  • 网站的互联网连接要求有多可靠？这对备份系统或冗余连接要求和测试有何影响？
  • 管理网站内容更新需要什么过程？
  • 维护、跟踪和控制页面内容、图形、链接等有哪些要求？
  • 将遵循哪种 HTML 规范？严格程度如何？目标浏览器允许哪些变体？
  • 整个网站或网站的某些部分对页面外观和/或图形是否有任何标准要求？
  • 如何验证和更新内部和外部链接？多久一次？它会发生吗？
  • 是否可以在生产系统上进行测试，还是需要单独的测试系统？
  • 如何考虑浏览器缓存、浏览器选项设置的变化、拨号连接的可变性以及实际的互联网”流量拥塞”问题进行测试？
  • 服务器日志记录和报告要求的范围或定制程度;它们是否被视为系统的一个组成部分，是否需要测试？
  • 如何维护、跟踪、控制和测试 CGI 程序、小程序、javascript、ActiveX 组件等？
  • 页面应最多为3-5个屏幕，除非内容高度集中在单个主题上。如果较大，请在页面中提供内部链接。
  • 页面布局和设计元素应在整个网站中保持一致，以便用户清楚地知道他们仍在网站上。
  • 页面应尽可能独立于浏览器，或者应根据浏览器类型提供或生成页面。
  • 所有页面都应具有指向该页面外部的链接;不应该有死胡同的页面。
  • 每个页面上都应包含页面所有者、修订日期以及指向联系人或组织的链接。

网站测试常见问题

  • 网站是否按预期运行？
  • 网站易于浏览吗？
  • 网站是否可以在不同设备上访问？
  • 网站是否足够安全？
  • 网站性能是否达到标准？
  • 在网站上输入的数据是否准确存储并跨会话保留？
  • 网站是否与工作流程中的其他界面很好地集成在一起？
  • 上线后，网站的性能是否仍能达到预期？

将详细介绍上述问题，以了解测试范围，并了解如何执行网站测试。

网站是否按预期运行？

要考虑的点

  • 用户导航到网站的不同页面并完成端到端工作流程
  • 如果用户可以选中/取消选中复选框
  • 如果用户可以从下拉字段中选择值
  • 如果用户可以选择/取消选择单选按钮
  • 不同的导航按钮，如提交，下一步，上传等按钮运行良好
  • 日历正确加载并允许用户选择日期
  • 计算正在按照实施方式进行
  • 搜索功能正在工作（如果有）
  • 正确的信息显示
  • 指向其他页面的各种内部和外部链接
  • 网页上字段的正确 Tab 键顺序
  • 应验证正输入和负输入的必填字段和可选字段
  • 应验证每个 Web 字段的默认值
  • 电子邮件功能是为网站上的某些操作而实现的

网站与搜索引擎兼容非常重要。因此，我们应该审查网站的HTML语法正确性，格式和合规性标准，如WS-I，ISO和ECMA。

考虑到用于维护登录会话的 Cookie，应通过启用/禁用 Cookie 或使用不匹配的域来测试网站。还可以通过重置 Cookie 将浏览器恢复到原始状态来跨会话执行测试。

QA还应验证网站cookie是否始终以加密格式存储在本地。

考虑到我们的电子商务网站，如果用户导航到预期的页面，则应单击并验证网页上可用的各种链接，例如男士时尚，女性时尚，儿童时尚，家居饰品，电子电器，书籍，电影和音乐等。

同样，不同的功能，如登录，注册，搜索选项，过滤器，排序订单，添加到购物车等，应该在不同的网页上进行验证，如登录页面，注册页面，产品详细信息页面，购物车，订单审查，付款等。应检查网站的会话/Cookie管理，如会话过期和会话存储等。

网站易于浏览吗？

要考虑的点

  • 网站内容应信息丰富，结构化和逻辑链接，以便用户可以轻松理解
  • 网页控件应易于用户导航
  • 网站应上传帮助和说明文档
  • 网站应具有搜索功能，以方便最终用户
  • 访问/从主菜单到所有页面都应该在那里
  • 应验证网站内容是否存在任何拼写错误
  • 网站应遵循背景颜色，图案，样式，字体，图像位置，框架，边框等上下文中定义的准则。
  • 考虑到来自不同国家/地区具有不同语言，货币等的用户可以访问该网站，因此该网站应习惯于翻译功能。

用户缩放和反射器是少数可用于执行可用性测试的工具。

电子商务网站应该是客户友好的，易于浏览和引人注目的。所有网页都应验证其可访问性、字体、样式、图像、拼写错误和产品相关信息。网站应配备相关的帮助文档和客户支持设施。

考虑到基于触摸屏的界面的增加，需要验证关键输入和触摸屏输入的可访问性。同样，应验证图像和网站内容在不同屏幕尺寸（手机、笔记本电脑和标签等）上的可用性。

网站是否可以在不同设备上访问？

假设网站可以被使用不同设备组的一系列用户访问，需要确保网站在所有这些设备上都能正常运行，而不会出现任何故障。

为了确保相同，应执行兼容性检查，这是兼容性测试附带的。在网站的兼容性测试期间，确保网站在不同的浏览器，操作系统和设备（如笔记本电脑，手机，平板电脑，打印机等）上运行良好。

浏览器兼容性（跨浏览器测试）

该网站应该可以很好地与不同的浏览器配合使用，如Microsoft Internet Explorer，Microsoft Edge，Firefox，Google Chrome，Safari和Opera。这些浏览器的所有活动版本都应使用打开/关闭的不同浏览器功能进行验证。

此外，在执行跨浏览器测试时，QA还应检查跨浏览器的最佳网站性能。

操作系统兼容性（跨平台测试）

为了识别潜在的用户体验问题，应该在Windows，Linux，Unix.MAC，Solaris等各种平台上测试网站，以确保操作系统的兼容性。

设备兼容性（跨设备测试）

网站可以通过不同的设备（如笔记本电脑，手机，平板电脑等）浏览，并提供不同的操作系统，如iOS，Android，Windows等。因此，测试也应该在设备上执行，涵盖以下方案。

  • 网站屏幕尺寸应根据设备进行调整
  • 设备应具有屏幕旋转功能
  • 网站不应在具有不同网络速度的不同设备上显示任何加载问题
  • 验证设备进入/退出网络范围时的网站行为
  • 验证网站在 CPU 和内存不足时的行为，以支持不同的外形规格

对于电子商务网站，兼容性检查是最重要的测试类型之一。客户群将很大，并将从不同的浏览器，操作系统和设备访问我们的网站。

考虑到移动平台越来越受欢迎，应该确保在可接受的加载时间内以小尺寸加载网站。验证不同网络速度的使用以确保它对所有客户都可用也很重要。

网站是否足够安全？

测试的点

  • 只有经过身份验证的用户才能访问网站
  • 网站用户应只能执行他们有权执行的任务
  • 应验证网站的 CAPTCHA 字段，以便用户识别
  • 在从安全页面移动到不安全页面时，应验证浏览器安全设置
  • Web 服务器保护应适用于无法访问的 Web 目录或文件
  • 确保在没有适当访问权限的情况下不应下载受限制的文件
  • 处于非活动状态的会话应在一段时间后自动终止
  • 应记录最终用户的所有无效和未经授权的尝试或间歇性的系统错误/故障，以便进行分析

漏洞管理，Veracode和SQL Map等工具可用于对网站执行安全测试。

网站性能是否达到标准？

要检查网站的性能，可以进行性能测试。它将评估应用程序在各种工作负载条件下的行为，这可能是一个现实的场景。如果系统在没有进行性能测试的情况下上线，则最终可能会出现系统运行缓慢或可用性差等问题，这可能会影响品牌形象和市场销售。

测试的点

  • 在正常和峰值负载条件下应观察网站
  • 应通过测量响应时间、速度、可扩展性和资源利用率来检查网站的性能
  • 如果系统在任何时间点发生故障或不稳定，则应使用解决方案进行适当的RCA（根本原因分析）
  • 应确定网络延迟问题（如果有）

在网站上输入的数据是否准确存储并跨会话保留？

数据库是 Web 应用程序的关键组件之一，它保存通过网站输入的完整信息。因此，应确保在没有任何操作的情况下将正确的用户数据保存在数据库表中，并在验证下保持数据完整性。

  • 验证跨用户界面（即网站 UI 和数据库）的数据一致性
  • 验证每当网站应用程序执行插入/更新/删除操作时，数据库表是否正确更新
  • 验证技术查询的响应时间，并根据需要对其进行微调
  • 检查数据库连接和访问权限

作为测试电子商务网站的QA团队成员，可以执行以下活动，并每次在相应的数据库表中验证更改。这将确保网站UI和DB都是一致的。

1）下订单购买产品。

2） 取消产品。

3）选择交换产品。

4） 选择退货。

网站是否与工作流程中的其他界面很好地集成在一起？

执行接口级测试以检查网站与不同接口（如Web服务器和数据库服务器）的平滑交互。

在接口测试期间，测试人员需要确保应用程序请求是否正确发送到数据库，并且正确的信息作为输出显示给客户端。Web 服务器不应在任何时间点引发任何拒绝异常，数据库应始终与应用程序保持同步。

上线后，网站的性能是否仍能达到预期？

一旦产品进入生产环境，应进行定期检查，以保持对质量控制的检查。

在生产中验证产品时，可以考虑以下情况

  • 应定期执行 Web 应用程序测试，并应将测试日志保存为符合服务级别协议 （SLA） 的证明
  • 应检查自动扩展系统和负载均衡器是否就位且正常运行
  • 检查最终用户体验，并尝试发现在 QA 测试期间通常未被注意到的缺陷或恶意攻击
  • 监控峰值负载期间的产品响应时间
  • 实时执行边缘级测试用例，以识别网络故障、连接故障或意外呼叫中断

不同类型的页面应该如何测试

发表于 2022-02-04 分类于 测试

网站可以分为许多类型，所有这些都可以缩小在静态和动态类型下。爱这里详细讨论4种类型及其测试方法。在此之前，对这些类型进行归类处理。

  • 静态网站测试
  • 动态 Web 应用程序测试
  • 电子商务网站测试
  • 移动网站测试

静态网站测试

一个静态网站将为在不同时间访问该网站的所有访问者显示相同的内容。它也被称为信息网站。在静态网站上，只有开发人员才能在代码中进行更改。这种类型的网站将没有任何主要功能，它完全取决于UI设计。

测试一个简单的静态网站非常容易，在测试时只需考虑几件事。将在下面提到：

测试点

#1）测试GUI设计是必须的，因为静态网站完全依赖于它。您需要将待测试的PSD文件与开发的网页进行比较。检查设计中的所有元素是否应显示在开发的页面上。

#2） GUI设计的另一部分是检查字体大小，字体样式，间距和颜色是否都已重现。

#3）其次，需要检查链接（页面链接）以查看它是否正常工作。另外，找出是否有不可以正常访问的链接？

#4）通过比较客户端提供的内容来验证所有网页中的拼写和内容。

#5） 在某些情况下，图像将无法正确显示，可能会损坏或有时图像被掩盖，可能会显示错误的图像。必须严格检查。因为对于一个静态的网站，只有内容和图像才会赋予生命。

#6）仔细检查滚动条，将面临的问题是出现不需要的滚动或滚动被隐藏（它可能会隐藏内容），问题一般出现于水平和垂直滚动轴。

#7）如果联系表单检查，则通过发送一些虚拟消息来正常工作。

在联系表格上要检查的事项是：

  • 邮件是否正确发送，是否显示成功邮件？
  • 检查收到的电子邮件是否按照设计的正确格式发送给相关人员？
  • 检查电子邮件不应该作为垃圾邮件落入垃圾邮件吗？
  • 如果有回复电子邮件触发器被激活，那么检查发件人是否收到了邮件？

#8）检查它是否是无差错的网页，并使用W3验证器或其他相关软件对其进行验证。

#9）在静态网站中要检查的一些常量内容，

  • 检查标签栏上是否存在图标
  • 网址应包含正确的网页标题
  • 如果存在版权信息，则应显示
  • 如果有联系表格，验证码是必须的。[它可以防止垃圾邮件]
  • 检查网站的加载速度。[静态网站不应花费太多时间进行加载]。如果在加载时使用了gif图像，则跟踪其功能

除此之外，还有大量的东西必须在每个网站的后端进行测试，包括系统测试，安全测试，接口测试，兼容性测试和性能测试等。为此，您要具备技术知识。在静态网站中，如果还需要进行功能测试，找不到更多功能。

动态Web应用程序[CMS网站]

用户可以定期更新和更改其网站内容的类型。从这里开始，我将使用”页面测试”这个词，而不是动态网站测试。页面程序是前端和后端编程的组合。

前端将是HTML和CSS，vue，而后端使用PHP，Javascript和ASP等编程语言。使用此后端，用户/客户端可以添加或更改网站上的内容。

测试页面比测试静态网站简单，但比测试电子商务网站的困难成都差不了多少。功能测试是测试 页面时要执行的最重要的事情。Web应用程序可能包含非常复杂的功能，因此测试人员在测试时需要非常小心。

有两种不同类型的Web应用程序，一种是用户在前端不会执行任何操作（即只有后端更改才会反映在前端），另一种是最终用户将在前端本身工作（例如登录，注册，新闻通讯订阅和其他类似操作）。因此，应根据它进行测试。

测试点

我在静态网站测试中提到的要点也将包括在测试Web应用程序时。除此之外，还需要注意以下事项。

#1）在GUI部分中，所有字段和按钮都必须使用提示工具，应正确完成字段对齐（间距），禁用的字段/按钮应灰显，字段/按钮应采用SRS中的标准格式，如果出现问题应显示错误消息，弹出消息应仅显示在网页的中心， 下拉菜单不应被截断。Tab 快捷键应该适用于所有字段等。

#2）在功能部分中，如果您的Web应用程序具有登录或注册功能，请检查必填字段验证，表单验证（即数字字段应仅接受数字，而不接受字母表），字段的字符限制（即只能输入这些字符）。

字段的特殊字符和负数限制，测试电子邮件功能，测试文档上传（即只能上传指定的文档类型），超时功能，排序功能，javascript在兼容浏览器上工作等应该进行测试。

#3）当进入后端功能部分时，测试图像上传是否有损坏的图像，在字段中输入的文本是否有效。后端更新应该反映在前端，数据库测试（即是否可以添加新字段或删除不需要的字段）所有这些操作都要执行。

对于页面（动态网站）来说，性能不是必需的，因为它的内容非常少。如果需要，可以使用您熟悉的工具。如果想做简单的性能测试，选择一些标准的在线性能工具

电子商务网站

与上述两个相比，电子商务网站有些复杂。测试人员在测试电子商务网站时需要非常谨慎。电子商务网站中有很多东西需要检查，只是涵盖了我在电子商务网站测试方面的一些经验问题。

在 GUI 部分中，需要检查 SRS 中的所有功能以及功能是否相同。对于所有商业网站，功能几乎相同。

功能方面，需要检查所有页面，例如主页（包括特色产品，特别优惠显示，登录详细信息，搜索功能），产品详细信息页面，类别页面，下订单，支付网关，一切都必须进行测试。

测试点

#1） 检查购买或增加数量时购物车是否正在更新。在所有页面和情况下检查此功能。

#2）检查特殊优惠券和优惠是否应用于正确的订单，并且看到折扣价格是否显示。

#3） 有时，在更新单个产品时，通过考虑产品中的变体数量，它会成倍增加。因此，请检查是否显示单个产品及其变体是否正确显示。（遇到过这个问题）

#4） 检查过滤器选项是否正常工作。如果根据所选的类别和定价进行过滤？

#5）注册时，应进行用户验证。只有新用户可以注册。

#6） 如果现有用户将产品添加到购物篮，则在下次登录期间，愿望清单部分也应保存并显示。

#7）比较产品应通过根据后端分配的某些规格比较产品来工作。

#8）检查货币转换器是否正常工作。根据所选的国家/地区，货币转换器应显示相关的价格和税率。

#9）通常许多插件都用于电子商务（WordPress和类似）网站，需要非常小心。插件安装可能会发生冲突或影响任何其他主要功能。因此，请跟进插件安装及其使用情况。

#10）检查社交共享选项是否适用于单个产品。

#11） 运费应根据所选区域生成。并检查税率生成。（在最终用户购买期间，可能会导致一些法律问题）。

#12）只有在提供有效的卡详细信息时，支付网关才应有效。验证应适用于卡号和CCV代码号。[最好在卡号字段本身上进行验证]。

#13）在购买过程中的每个过程中都应生成电子邮件（注册，产品订购，付款成功，订单取消，订单收到和其他电子邮件触发器（如果有）。

#14）检查实时聊天与一些垃圾电子邮件。

注意：通常，电子商务网站不会为移动兼容性而开发，并且在进入移动版本时将生成应用程序。在某些情况下，他们不会创建应用程序，而是创建一个移动兼容的网站。在这种情况下，需要仔细检查是否有任何缺少的功能和UI偏差。

这些是我在测试电子商务网站时面临和注意到的一些问题。除此之外，还需要检查与电子商务网站相关的所有一般事项。

手机网站

首先，要明确什么是一个移动网站，通常人们认为移动网站和移动应用程序是相同的，但实际上，移动网站是使用HTML页面开发的，并且只能通过互联网连接进行查看。

但是移动应用程序只不过是一个应用程序，可以在没有互联网连接的情况下下载和使用。在这里，我们中的许多人感到困惑并提出一个问题 移动网站和响应式网站有什么区别？

响应式网站意味着使内容适合移动设备大小，而不是创建版本，而移动网站正在创建一个不是反射桌面版本的新版本。在移动网站中，将只有有限的页面，并且不需要的功能将在此处删除。

测试移动网站比其他类型的网站有些乏味。它将具有单独的设计，在测试功能时需要小心。

测试点
  • 通常，将使用模拟器来测试移动网站，可以获得理想的结果，但是希望您在真实设备上进行测试。我在真实设备（尤其是苹果设备）中进行测试时，遇到了许多问题。实际设备规格可能与开发的网页冲突。
  • GUI和可用性测试更重要，因为它不是桌面版本的反映。
  • 性能是移动网站测试需要考虑的另一个重要因素。在实际设备中进行测试时，可以跟踪与性能相关的问题。
  • 检查从移动设备浏览正常网页链接是否被移动链接触发。
  • 检查移动网站上的页面滚动，页面导航，文本截断等。

B/S架构如何测试

发表于 2022-02-04 分类于 测试

随着信息技术的发展，人们越来越习惯与在网上搜索信息来做出决定，因此网站已经是各类企业的必备之选。但是仅仅拥有一个网站还不够，还需要提供一个信息丰富、可访问和用户友好的网站。为了实现这样的功能，网站要经过良好的测试，这样的测试测试成为B/S架构测试即页面测试。

什么是页面测试？

页面测试是软件测试中的一种，用于测试网站或者web类应用程序是否存在一些问题。在上线之前要对基于页面的应用程序进行完整测试。

基于页面的系统在面向最终用户上线之前，需要进行端到端的全面的测试。

通过页面测试，可以确保基于页面的系统正常运行，且可以被用户接受，UI测试和功能测试是页面测试的首选。

常见的页面测试类型

  • 功能测试
  • 可用性测试
  • 接口测试
  • 兼容性测试
  • 性能测试
  • 安全测试
功能测试

测试页面中所有的链接，数据库连接，页面中用于提交或获取用户信息的表单，Cookie测试等

链接测试
  • 测试从所有页面到受测特定域的传出链接。
  • 测试所有内部链接。
  • 测试链接跳转到同一页面。
  • 测试链接用于从网页向管理员或其他用户发送电子邮件。
  • 测试以查看是否存在任何孤立页面。
  • 最后，链接检查包括，检查上述所有链接中的断开链接。
表单测试

表单在任何网站都是不可或缺的，一般用于接收来自用户信息。如何进行表单测试呢？

  • 首先，检查每个字段的所有验证。
  • 检查字段中的默认值。
  • 表单中对表单中字段的错误输入。
  • 用于创建表单的选项（如果有）、表单删除、查看或修改表单。
cookie测试

Cookie 是存储在用户计算机上的小文件。这些基本上用于维护会话 - 主要是登录会话。通过在浏览器选项中启用或禁用 Cookie 来测试应用程序。

在写入用户计算机之前，测试 Cookie 是否已加密。如果您正在测试会话 Cookie（即在会话结束后过期的 Cookie），请在会话结束后检查登录会话和用户统计信息。通过删除 Cookie 来检查对应用程序安全的影响。

HTML/CSS测试

如果您正在为搜索引擎优化您的网站，那么HTML / CSS验证是最重要的。主要验证网站的HTML语法错误。检查该网站是否可被不同的搜索引擎抓取。

数据库连接测试

数据一致性在 Web 应用程序中也非常重要。在编辑、删除、修改表单或执行任何与数据库相关的功能时检查数据完整性和错误。

检查是否正确执行了所有数据库查询，检索了数据并正确更新了数据。有关数据库测试的更多信息可能是数据库上的负载，我们将在下面的Web负载或性能测试中解决此问题。

在测试网站功能时，还应该测试以下内容：

链接

i. 内部链接
ii.外部链接
iii.邮件链接
iv.断开的链接

表格
  1. 字段验证
  2. 错误输入的错误消息
  3. 可选和必填字段
数据库

测试将针对数据库完整性进行。

可用性测试

可用性测试是测量系统的人机交互特性，并确定不足以进行纠正。

导航测试

导航是指用户如何浏览网页、不同的控件（如按钮、框）或用户如何使用页面上的链接浏览不同的页面。

可用性测试应包括以下内容
  • 该网站应易于使用。
  • 所提供的说明应非常清楚。
  • 检查提供的说明是否完全符合其目的。
  • 每个页面上都应提供主菜单。
  • 它应该足够一致。
内容检查

内容应符合逻辑且易于理解。检查拼写错误。深色背景使用会让用户不喜欢，不应在网站主题中使用。应该遵循一些用于网页和内容构建的标准颜色。这些是普遍接受的标准。内容应该是有意义的。所有锚文本链接都应正常工作。图像应以适当的尺寸正确放置。这些是页面开发中应遵循的一些基本重要标准。你的任务是验证所有内容以进行 UI 测试。

用于帮助用户搜索信息

与搜索选项一样，站点地图也有助于处理文件等。站点地图应与网站上的所有链接一起提供，并具有正确的导航树视图。检查站点地图上的所有链接。

“在网站中搜索”选项将帮助用户轻松快速地找到他们正在寻找的内容页面。这些都是可选项目，如果存在，则应对其进行验证。

接口测试

测试方法

发表于 2022-01-29 分类于 测试

图片是网上找的。测试方法对于所有类型的测试都很常见，不仅因为它是大数据或Hadoop测试，当我们进入正常的手动测试或自动化测试或安全测试，性能测试时也是如此，因此任何类型的测试都遵循相同的方法。

需求

作为测试方法的一部分，我们需要从需求开始，需求是一个基本的东西，现在在敏捷过程中，我们称之为故事和史诗。史诗只是一个更大的要求，而故事是较小的要求。

需求基本上包含所有的数据模型，目标，源以及我们需要应用哪种转换，我们必须使用哪种工具？所有这些类型的详细信息都将在要求中提供。

这基本上是客户要求或客户要求。基于此要求，我们将开始测试过程。

时间预估

方法的另一部分是时间预估，作为测试的一部分，我们需要花费多少时间来完成整个活动。我们做测试计划，准备测试场景，准备测试用例和执行，我们将发现缺陷并报告它们并准备测试报告。

所有这些活动都需要一些时间，因此需要多少时间来完成所有这些活动，这基本上称为时间预估。我们需要给管理层一些粗略的估计。

测试计划

测试计划只不过是关于流程的描述，要测试什么，不要测试什么，测试的范围是什么，时间表是什么，需要多少资源，硬件和软件要求以及将使用的时间表和测试周期是什么，我们需要的测试级别是什么等。

在测试计划期间，他们将对项目进行某些资源分配，我们拥有哪些不同的模型，需要多少资源以及需要什么样的技能集等，所有这些内容和方面都将包含在测试计划阶段中。大多数情况下，领导级别或管理层人员将执行测试计划。

测试方案和测试用例

完成测试规划后，我们需要准备测试方案和测试用例，特别是对于大数据测试，我们需要一些文档以及需求文档。除了这个要求文件，我们还需要什么？

我们需要包含客户需求的需求文档，同时我们需要输入文档，即数据模型。数据模型在某种意义上什么是数据库架构，什么是表以及什么是关系所有这些数据将在数据模型中可用。

此外，我们有映射文档，例如在关系数据库中，我们有一些表，在HDFS的数据仓库中通过ETL加载数据后，我们需要做什么？即映射数据类型。

我们怎么知道，我们需要验证的所有映射是什么？因此，这些映射将出现在映射文档中。在映射文档中，客户将提供各种映射。

此外，我们还需要一个设计文档，开发团队和QA团队都需要的设计文档，因为在设计文档中，客户将提供，他们将实现哪种类型的Map Reduce作业，什么类型的MapReduce作业需要输入以及什么类型的MapReduce作业提供输出。

同样，如果我们有HIVE或PIG，客户创建的所有UDF是什么，以及他们将采取的所有输入以及他们将产生什么样的输出等。

要准备测试方案和测试用例，我们需要手动准备所有这些文档：

  • 需求文档
  • 数据模型
  • 映射文档
  • 设计文档

这些文件可能因组织而异，并且没有强制性规则规定我们必须拥有所有这些文件。有时我们拥有所有文档，有时我们只有两到三个文档，或者有时我们还需要依赖一个文档，这取决于项目复杂性，公司时间表和所有内容。

评审测试方案和测试用例

我们需要对测试方案和测试用例进行审查，因为不知何故或在某些情况下，我们忘记或错过了一些测试用例，因为每个人都无法想到可以使用需求完成的所有可能的事情，在这种情况下，我们需要从第三方工具或其他人那里获得帮助。

因此，每当我们准备一些文档或执行某些操作时，我们都需要有人来审查来自同一团队的内容，例如开发人员，测试人员。他们会给出适当的建议，以包含更多内容，或者还建议更新或修改测试方案和测试用例。

他们提供所有注释，在此基础上，我们将更新我们的测试方案和测试用例以及我们需要在整个团队中发布的文档的多个版本，直到文档按照要求完全更新为止。

测试执行

文档准备就绪后，我们将从上级团队获得签核，以开始基本上称为测试用例执行的执行过程。

如果我们想在执行期间执行测试用例，我们需要检查开发人员是否必须发送信息，如果是正常的功能测试或其他一些测试或自动化测试，我们需要构建。但是，从Hadoop或BigData Testing的角度来看，开发人员将提供MapReduce Jobs。

HDFS文件 - 在HDFS中复制的任何文件，这些文件信息都需要检查权限，由开发人员创建的HIVE脚本以验证HIVE表中的数据，并且我们还需要由开发人员开发的HIVE UDF，PIG脚本和PIG UDF。

这些都是我们需要从开发人员那里得到的东西。在执行之前，我们应该拥有所有这些东西。

对于MapReduce Jobs，他们将提供一些JAR文件，并且作为HDFS的一部分，他们已经在HDFS中加载了数据，并且文件应该准备就绪，并且HIVE脚本可以验证HIVE表中的数据。无论他们实施了什么UDF，都将在HIVE UDF中提供。对于 PIG 脚本和 UDF，我们也需要同样的东西。

缺陷报告和跟踪

一旦我们执行了测试用例，我们就会发现一些缺陷，一些预期的和一些实际的不等于预期的结果，所以我们需要列出相同的缺陷，并将它们提供给开发团队解决，这基本上称为缺陷报告。

假设如果我们在MapReduce作业中发现一些缺陷，那么我们将将其报告给开发人员，他们将再次重新创建MapReduce作业，并进行一些代码级修改，然后他们将再次提供最新的MapReduce作业，我们需要对其进行测试。

这是一个持续的过程，一旦作业经过测试并通过，我们再次需要重新测试它并将其报告给开发人员，然后获取下一个用于测试的工作。这就是缺陷报告和跟踪活动的完成方式。

测试报告

一旦我们完成了所有的测试过程并且缺陷已经关闭，那么我们需要创建我们的测试报告。测试报告是我们迄今为止为完成测试过程所做的一切。所有的规划，测试用例的编写和执行，我们得到的输出等等，一切都以测试报告的形式记录在一起。

我们需要每天或每周或根据客户的需求发送这些报告。如今，组织正在使用敏捷模型，因此每个状态报告都需要在每日Scrums期间更新。

大数据测试-测试类型

发表于 2022-01-27 分类于 大数据

之前的文章写了大数据的存储或处理的一些程序，对于后续的这篇大数据如何测试，却迟迟无法落笔。因为每个公司都是根据自己的业务来使用大数据平台的一个或者多个功能的。这里随便聊下大数据测试！（就是想到哪里写到哪里）

大数据测试与传统测试的区别

传统测试：特定条件下运行的结果进行观察或者验证，构建上下文验证输出

大数据测试：数据正确，系统正确仅仅是第一步，后边还有诸多额外工作，如数据计算后结果，系统性能等。

大数据测试的挑战

  1. 复杂的数据流，数据的多样性；
  2. 集群规模大，数据量大
  3. 复杂算法测试
  4. 无明确的验收标准

大数据测试的测试类型

架构测试

进行此测试是为了确保数据处理正确并符合要求。实际上，Hadoop系统处理大量数据，并且资源非常全面。如果体系结构不正确，则可能会降低性能，因此数据处理可能会中断并可能发生数据丢失。

数据库测试

需要确保从源数据库或本地数据库获取的数据必须是正确和适当的。需要检查源数据库中可用的数据是否与在Hadoop系统中输入的数据匹配。

需要验证Hadoop系统中的数据在处理后或转换后是否正确，并通过适当的验证和确认加载到客户端环境中。作为数据库测试的一部分，需要完成CRUD操作，即在本地数据库中创建数据，然后检索数据，需要搜索它，它应该在加载到数据仓库之前和之后在数据库中可用，并从数据仓库到客户端的环境。在存储或加载和处理数据的每个阶段验证任何更新的数据。删除任何损坏的数据或任何重复和空的数据。

性能测试

作为性能测试的一部分，我们需要检查数据的加载和处理速度，例如IOPS（每秒输入输出）。

需要检查输入数据或数据作为输入的速度，从各种数据库到数据仓库或Hadoop系统，以及从Hadoop系统或数据仓库到客户环境。

还必须检查来自各种数据库的数据以及作为输出的数据仓库的速度。这就是我们所说的每秒输入输出或 IOPS。

除此之外，另一个方面是检查数据吸收和数据分发的性能，以及数据仓库从各种数据库和客户端系统从Hadoop系统消耗数据的速度。

此外，作为测试人员，我们需要检查数据分发的性能，例如，数据以多快的速度分发到Hadoop系统或数据仓库中可用的各种文件。同样，在将数据分发到客户端系统时也会发生相同的过程。

Hadoop系统或数据仓库由多个组件组成，因此测试人员需要检查所有这些组件的性能，例如MapReduce作业，数据插入和使用，查询的响应时间及其性能以及搜索操作的性能。所有这些都包含在性能测试中。

功能测试

功能测试包含对所有子组件，程序和脚本，用于执行存储或加载和处理操作的工具等的测试。对于测试人员来说，这是需要过滤数据的四个重要类型和阶段，以便客户端获得完美且无差错的数据。
